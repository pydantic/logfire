# Anthropic Streaming Usage Attributes

**Code-level architecture is in [code-spec](code-spec.md).**

**Every Anthropic streaming span must have the same usage attributes as non-streaming spans: `gen_ai.usage.input_tokens`, `gen_ai.usage.output_tokens`, `operation.cost`, and `gen_ai.usage.raw`.**
Context: Non-streaming `on_response()` already uses `get_anthropic_usage_attributes()` (in `anthropic.py`) which delegates to the shared `get_usage_attributes()` (in `usage.py`). Anthropic streaming `get_attributes()` currently ignores usage entirely — the streaming log span has no token counts, no cost, and no raw usage. Non-streaming Anthropic and all OpenAI paths are already handled.

**Usage must be extracted from the final accumulated message, not partial streaming snapshots.** *(from "Every Anthropic streaming span")*
Context: `get_usage_attributes()` passes `response.model_dump()` to `genai_prices.extract_usage()`, which extracts both the model and usage from the serialized response. Anthropic charges for server-side tools (e.g., web search at $10/1,000 searches) tracked via `usage.server_tool_use`, and output tokens (more expensive than input tokens). Both can change during streaming — server-side tool execution can happen mid-response, and output tokens grow as tokens stream out. Unlike OpenAI (where `genai_prices` may need to count tool calls in output messages), everything Anthropic needs for pricing is in the `Usage` object — but only if it reflects the final state.
A response object with stale usage would produce wrong costs. The `message_start` event's usage is an initial snapshot that may have incomplete `output_tokens` and `server_tool_use`; only after all chunks are processed is the usage final.

**`record_chunk()` uses the Anthropic SDK's `accumulate_event()` to build a complete `Message` from streaming events.** *(from "Usage must be extracted from the final accumulated message")*
After all chunks, the snapshot is a complete `Message` with accurate final usage — the same quality of object that non-streaming `on_response()` receives.
Context: The Anthropic SDK exports `accumulate_event(event, current_snapshot)` in `anthropic.lib.streaming._messages`. It takes a raw stream event and the current message snapshot, returning an updated `ParsedMessage` (a `Message` subclass). It handles all event types: `message_start` initializes the message with model, usage, and id; content block events accumulate text and tool use; `message_delta` updates `stop_reason` and all usage fields (`output_tokens`, `input_tokens`, cache tokens, `server_tool_use`) with cumulative final values. We investigated whether to use the SDK's high-level `MessageStream` class (which calls `accumulate_event` internally) instead of the `StreamState` abstraction. This isn't possible because logfire instruments at the `client._request` level, where `stream_cls` is always the raw `Stream[RawMessageStreamEvent]`. The high-level `MessageStream` wraps this raw stream as an outer layer (in `MessageStreamManager.__enter__`) — we don't subclass it. Calling `accumulate_event` directly gives us the same accumulation without requiring architectural changes to the streaming infrastructure.

**`get_anthropic_usage_attributes()` is reused for streaming — the accumulated message is a complete `Message`.** *(from "record_chunk() uses the Anthropic SDK's accumulate_event()")*
Since `accumulate_event` produces a `Message` with final usage, we can call `get_anthropic_usage_attributes(self._message)` in `get_attributes()` — the same function used by non-streaming `on_response()`. No separate streaming-specific usage extraction is needed. This also means `gen_ai.usage.raw` reflects the final accurate usage (including updated `output_tokens` and `server_tool_use`), and cost calculation gets a complete response object. This goes against the initial design (which had streaming call `get_usage_attributes()` directly with manually-extracted values) because the original approach couldn't produce accurate data for cost calculation from partial snapshots.

**The manual text accumulation (`_content` list, `content_from_messages()`) is replaced by reading from the accumulated message.** *(from "record_chunk() uses the Anthropic SDK's accumulate_event()")*
Currently `AnthropicMessageStreamState` manually extracts text deltas into a `_content` list using the `content_from_messages()` helper. Since `accumulate_event` already accumulates content blocks on the message, `get_response_data()` and `get_attributes()` can read text from `self._message.content` instead. This eliminates the redundant manual accumulation. Context: The `content_from_messages()` function in `anthropic.py` may still be needed by other code — check before removing.

**No try/except around the usage call in `get_attributes()`.** *(from "get_anthropic_usage_attributes() is reused for streaming")*
`get_usage_attributes()` has fine-grained internal error isolation (tokens, raw, and cost each fail independently). In the streaming path, `get_attributes()` is called inside `record_streaming()` which creates a log span — errors propagate but don't crash the user's code. Adding a broad `try/except` would hide real bugs. This matches the non-streaming approach and the OpenAI streaming approach.

**Existing streaming behavior (response_data, output messages, duration) must not change.** *(from "Every Anthropic streaming span")*
This is purely additive — new usage attributes are added alongside existing ones in the dict returned by `get_attributes()`. Existing streaming tests will gain the new usage attributes in their snapshots; Bedrock streaming tests follow the same pattern.

**Usage attributes are set regardless of semconv version.** *(from "Every Anthropic streaming span")*
This matches the existing non-streaming pattern (where `on_response()` sets usage attributes outside version-specific branches) and the OpenAI streaming pattern. Context: The streaming code has version branching (`SemconvVersion`) — version 1 sets `response_data`, `'latest'` sets `OUTPUT_MESSAGES`. Usage attributes are set unconditionally, outside version-specific branches.
